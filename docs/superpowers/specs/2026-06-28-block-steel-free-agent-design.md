# Блокировка свободного агентства для игроков команды СТИЛ

**Дата:** 2026-06-28
**Статус:** согласовано, готово к плану реализации

## Задача

Обратная связь от руководителя клуба: игроки команды **СТИЛ** не должны
иметь возможности стать свободными агентами. Если игрок СТИЛ пытается это
сделать, бот должен:

1. **Заблокировать** действие — карточка НЕ появляется на доске агентов.
2. Показать игроку ошибку:
   `Невозможно стать свободным агентом, так как вы являетесь игроком команды СТИЛ.`
3. Отправить **уведомление администратору** (ADMIN_TG_ID = 1380368896) с ФИО
   игрока, который пытался это сделать.

## Что считается «стать свободным агентом»

Активация карточки — переход `active = 1`. Именно в этот момент карточка
появляется в поиске агентов. Кнопки:
- Mini App: «Заявить себя свободным агентом» / «Добавить в поиск агентов»
  → `POST /api/card/status` с `{active: 1}`.
- Бот (legacy): callback `become_agent` → `become_agent_callback`.

Под-статус «🔍 Ищу команду» (`looking`) отдельно НЕ блокируется: он виден
только после активации, а активация для игроков СТИЛ невозможна.

## Определение игрока СТИЛ

По полю `current_team` карточки. Его заполняет парсер lfl.ru при импорте
профиля (`current_team = "" if is_free_agent else current_club`), то есть это
**текущий клуб** игрока на сайте. В карточке команда записана как `СТИЛ`.

Сравнение без учёта регистра, по нормализованной строке и по словам:
нормализуем (`strip`, `casefold`, схлопывание пробелов), затем считаем игрока
ограниченным, если нормализованная строка целиком равна `стил` **или** одно из
слов (разделённых пробелами) равно `стил`.

- `СТИЛ`, `стил`, `ФК Стил` → заблокировано.
- `Бастилия`, `Стиляги` → НЕ заблокировано (целое слово не равно `стил`).

Дубля/резерва у команды нет — дополнительных вариантов не добавляем.

### Точка 1 — Mini App: `api.py` → `/api/card/status`

В обработчике `card_status`, до выполнения активации, добавить проверку:
если `req.active == 1`, игрок ещё не активен (`not was_active`) и
`is_restricted_team(existing.get("current_team"))`:

- НЕ вызывать `activate_agent`, НЕ ставить уведомление командам;
- запустить фоновую задачу уведомления администратора;
- вернуть `{"ok": False, "error": "Невозможно стать свободным агентом, так как вы являетесь игроком команды СТИЛ."}`.

Фронт (`webapp/index.html`, `setStatus`) уже показывает `res.error` через
`tg.showAlert(res.error)` — изменения фронта не требуются.

Добавить в `api.py` загрузку `ADMIN_TG_ID`:
`ADMIN_TG_ID = int(os.getenv("ADMIN_TG_ID", "0") or 0)`. Если он не задан (0) —
уведомление просто не отправляется (тихо). Отправка через существующий
`_send_telegram`.

### Точка 2 — бот (legacy): `handlers/card.py` → `become_agent_callback`

Клавиатура бота урезана до `[🆘 Помощь]`, но callback `become_agent` ещё
зарегистрирован и встречается в карточных вью. Для надёжности закрываем и здесь.

`become_agent_callback` имеет две ветки:
- **свежий импорт** (`profile` из `context.user_data["multi_profile"]`): команда =
  `"" if profile.is_free_agent else profile.current_club`. Проверять
  `is_restricted_team(profile.current_club)` при `not profile.is_free_agent`.
- **активация существующей карточки**: проверять
  `is_restricted_team(agent.get("current_team"))`.

При срабатывании: НЕ активировать/НЕ сохранять как `active=1`, показать
`query.answer(<тот же текст ошибки>, show_alert=True)`, уведомить
администратора (`from handlers.admin import ADMIN_TG_ID`,
`context.bot.send_message`). Уведомление — обычный текст (без Markdown, чтобы
спецсимволы в имени не ломали отправку).

## Уведомление администратору

Текст (plain-text, без Markdown):

```
🚫 Игрок СТИЛ пытался стать свободным агентом

👤 {ФИО из карточки}
📱 {@username или tg_id}
🔗 {ссылка lfl.ru, если есть}
```

- ФИО — поле `name` карточки (в боте при свежем импорте — `profile.name`).
- Контакт — `@username` из initData/Telegram-аккаунта, иначе `tg_id`.
- Ссылка — `lfl_url` карточки, если непустая (в строке пропускаем, если пусто).

## Тесты (`tests/test_card_api.py`)

- Игрок СТИЛ (`current_team = "СТИЛ"`) вызывает `/api/card/status` `{active:1}`
  → ответ `ok == False` с нужным текстом ошибки; карточка осталась `active == 0`;
  уведомление командам (`_notify_teams_new_player`) НЕ поставлено.
- Обычный игрок (`current_team = "Динамо"`) → активируется как раньше
  (`ok == True`, `active == 1`).
- Юнит-тест `is_restricted_team`: `"СТИЛ"`, `"стил"`, `"ФК Стил"` → True;
  `"Бастилия"`, `""`, `None`, `"Динамо"` → False.

Уведомление администратору в тестах можно замокать/проверить вызов
`_send_telegram` (как уже делается для других уведомлений в тест-сьюте).

## Деплой

- Коммит + пуш в `origin/master` (правило авто-коммита для проекта).
- На сервере: `cd /opt/lfl-bot && git pull` + перезапуск `lfl-api` и `lfl-bot`
  (правится код и API, и бота). Новых колонок БД нет — миграции не требуются.

## Вне рамок / известное ограничение

- Если игрок СТИЛ **вручную сотрёт/переименует** команду в поле карточки, он
  сможет активироваться — детект только по тому, что показывает парсер. Это
  согласовано: ловим тех, у кого текущий клуб = СТИЛ.
- Реестр официальных составов команд не вводится (YAGNI). Расширение на другие
  команды — через `RESTRICTED_TEAMS` в `restrictions.py`.
